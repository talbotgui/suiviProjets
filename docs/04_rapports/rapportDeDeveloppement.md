## Étape 0 — Bootstrap du poste de développement et de l'outillage

### Codeur

Le dépôt, jusqu'ici exclusivement documentaire, a été doté d'un socle de code compilable et testable, conformément à l'objectif de cette phase et sans qu'aucun écran, aucune commande métier ni aucun connecteur réel n'ait été introduit à ce stade.

L'interface a été générée à l'aide du générateur officiel le plus récent de l'écosystème, en configurant dès la génération le routage, les feuilles de style, ainsi que l'intégration d'un outil d'analyse statique dédié à ce framework. Le cœur natif a été initialisé à l'aide de l'outil officiel de génération de squelette d'application de bureau, en conservant la disposition standard où l'interface se trouve à la racine du dépôt et le cœur natif dans un répertoire dédié, conformément au découpage en couches déjà arrêté dans la documentation d'architecture. Les répertoires structurants attendus, aussi bien côté interface que côté cœur natif, ont été créés, chacun accompagné d'une courte description de son rôle futur afin de rester suivi par le gestionnaire de versions malgré l'absence de code métier à ce stade.

La chaîne d'outillage a été figée : la version du langage d'interface et celle du langage natif installées dans cet environnement ont été verrouillées explicitement, de même que l'édition du langage natif, portée à la dernière édition stable disponible pour le compilateur installé. Les fichiers de verrouillage des dépendances des deux piles ont été générés puis utilisés en mode strict pour toutes les vérifications ultérieures.

Le formatage automatique et l'analyse statique ont été configurés pour les deux piles, avec un niveau d'exigence élevé côté interface : visibilité explicite systématique, type de retour explicite, documentation systématique de toute classe et de toute méthode, interdiction du type dynamique non typé, interdiction des assertions de type non justifiées, exhaustivité du traitement des discriminants de résultat, gestion explicite de toute promesse asynchrone, et surtout l'interdiction de toute fonction déclarée en dehors d'une classe, contrainte propre à ce projet ne correspondant à aucune règle générique existante et qui a donc nécessité l'écriture d'une règle dédiée. Une divergence assumée par rapport à la recommandation par défaut de l'outillage a été nécessaire pour autoriser les classes utilitaires ne comportant que des membres partagés par l'ensemble des instances, faute de quoi la convention « aucune fonction hors classe » n'aurait pas pu être respectée pour les fonctions utilitaires ; ce choix a été documenté directement dans le fichier de configuration concerné. Côté cœur natif, l'interdiction de tout code non sûr au niveau de chaque unité de compilation, l'interdiction des arrêts brutaux du programme en cas d'erreur, ainsi que la visibilité la plus restrictive possible ont été activées.

Un dispositif d'automatisation a été mis en place pour exécuter le formatage et l'analyse statique appropriés juste après chaque modification de fichier de code, humaine ou assistée par l'IA, en se basant sur l'extension du fichier concerné ; ce dispositif reste un confort local, l'intégration continue restant le niveau de vérité de référence pour l'analyse statique. Son fonctionnement a été vérifié concrètement en modification réelle, pas seulement par une relecture visuelle de sa configuration.

Un second dispositif d'automatisation, indépendant du premier, consigne l'horodatage et le contenu de chaque échange significatif directement soumis par l'utilisateur dans un journal strictement local, exclu du suivi de version et sans aucune transmission distante ; les échanges internes propres à une délégation de tâche à un agent auxiliaire en sont explicitement exclus, conformément à la décision déjà actée dans la documentation du poste de développeur.

Un fichier de référence exemplaire a été créé pour chacun des quatre types de fichiers récurrents identifiés par les normes de développement : un composant d'interface réutilisable, un service porteur d'un état réactif, une classe utilitaire à membres partagés, et un module du cœur natif dédié à l'interrogation d'une source externe. Chacun est explicitement identifié comme gabarit de référence non destiné à un usage fonctionnel réel, respecte intégralement les règles de rigueur retenues, et a été placé dans le répertoire structurant qui accueillera le code fonctionnel correspondant lors des phases suivantes. Chacun de ces gabarits est accompagné d'au moins un test automatisé démontrant son bon fonctionnement, plutôt que de rester une simple illustration statique.

La configuration du cadre de test de l'interface a été mise en place avec le framework retenu par les normes du projet, différent de celui proposé par défaut par le générateur d'interface le plus récent ; ce choix a nécessité un ajustement du mécanisme d'initialisation de l'environnement de test, l'application générée ne reposant plus, dans sa version la plus récente, sur le mécanisme de détection de changements historiquement associé à ce framework d'interface. Les seuils de couverture définis par les normes de tests ont été configurés par avance selon le périmètre concerné par chaque répertoire structurant, sans pour autant faire échouer les vérifications courantes tant qu'aucun code métier réel n'existe dans ces répertoires ; ce choix temporaire est documenté explicitement dans le fichier de configuration concerné, avec le rappel qu'il prendra effet automatiquement dès l'ajout de code réel, sans intervention supplémentaire.

Un squelette de pipeline d'intégration continue a été mis en place, strictement limité à la récupération du code et à l'installation de l'outillage des deux piles, sans aucune étape de vérification fonctionnelle, ces dernières étant explicitement différées à une phase ultérieure dédiée à l'intégration continue complète.

Les autres éléments d'outillage du poste de développement demandés à cette phase ont également été mis en place : le gabarit des variables d'environnement locales, le réglage de l'éditeur de référence pour le formatage à l'enregistrement et la validation continue par l'analyse statique, ainsi que l'exclusion, dans la configuration de l'assistant IA, de toute lecture directe des futurs fichiers de données de test chiffrés.

Toutes les vérifications exécutables dans cet environnement ont été rejouées jusqu'à obtenir un résultat entièrement conforme : compilation du cœur natif, exécution de sa suite de tests, formatage et analyse statique du cœur natif, compilation de l'interface, exécution de sa suite de tests, et analyse statique de l'interface. Une limite assumée de cet environnement a empêché toute vérification visuelle de l'ouverture effective d'une fenêtre applicative, faute de serveur graphique disponible ; cette vérification reste à réaliser par l'utilisateur sur son propre poste avant validation définitive de cette phase.

Un doute a été rencontré sur le choix précis de la version du générateur d'interface à utiliser : la version la plus récente disponible s'est révélée incompatible avec la version du langage d'exécution déjà installée et imposée par ailleurs, pour une raison tenant à une vérification interne du générateur lui-même plutôt qu'à une contrainte de compatibilité déclarée. La version immédiatement précédente, elle-même très récente, a donc été retenue à la place, ce choix étant documenté comme une décision explicite plutôt qu'une simple contournement silencieux.

Un doute a également été rencontré sur l'identifiant applicatif unique et sur la portée exacte de la politique de sécurité du contenu embarqué à renseigner dans la configuration du cœur natif, aucun des deux n'étant précisé dans la documentation existante ; des valeurs raisonnables ont été retenues à titre provisoire, à confirmer explicitement par un humain avant toute publication réelle.

Enfin, un fichier isolé, sans rapport apparent avec les actions menées dans le cadre de cette phase, a été constaté dans le dépôt en cours de session ; son origine n'a pas pu être établie avec certitude et il n'a pas été modifié, dans l'attente d'un examen humain.

Deux remarques formulées par un humain après relecture de l'ensemble de cette phase ont ensuite été intégrées avant validation définitive. La première a consisté à faire apparaître explicitement, dans le rangement des répertoires structurants de l'interface encore vides, une distinction jusque-là seulement implicite entre les modules ne conservant aucun état entre deux sollicitations et ceux en conservant un ; les cinq répertoires concernés ont été répartis selon cette distinction, sans que le rôle, le contenu ni le sous-regroupement fonctionnel de chacun n'en soient modifiés, et la documentation normative de référence a été mise à jour en premier, avant tout changement de code, conformément à l'ordre habituel de travail de ce projet.

La seconde a consisté à constater qu'un même mécanisme d'exclusion du gestionnaire de versions existait en double, une fois à la racine du dépôt et une fois dans le répertoire dédié au cœur natif, redondance non conforme à la pratique standard d'un seul mécanisme de ce type par dépôt ; l'exclusion propre au cœur natif encore absente de la racine a été reportée dans le mécanisme racine, puis le mécanisme redondant a été supprimé, après vérification qu'aucun outil du cœur natif ne dépendait de sa présence locale pour fonctionner.

### Relecteur

L'intégralité des éléments produits durant cette phase a été relue dans un contexte isolé de celui du développeur du code, en confrontant chaque choix aux documents normatifs déjà validés plutôt qu'aux seules intentions déclarées, et chaque vérification annoncée comme entièrement conforme a été rejouée concrètement plutôt que supposée acquise sur la seule foi du récit précédent.

Le fichier isolé signalé par le développeur du code a été examiné en détail. L'une des deux affirmations qu'il portait s'est révélée inexacte au moment de la relecture, l'élément qu'elle décrivait n'existant plus ou n'ayant jamais existé sous la forme rapportée ; l'autre affirmation était exacte mais décrivait une redondance mineure, déjà pour partie justifiée par une différence de portée entre les deux éléments concernés, jugée sans conséquence et volontairement conservée en l'état. Ce fichier ne figurant dans aucune liste de livrables documentée et ne relevant d'aucun propriétaire ni d'aucune durée de vie définie, il a été supprimé plutôt que corrigé sur place ; une règle a été ajoutée pour qu'un constat de cette nature soit désormais consigné directement dans ce rapport ou transformé en évolution de règle, jamais laissé sous la forme d'un fichier de note isolé.

Le dispositif d'automatisation local de formatage et d'analyse statique s'est révélé incomplet : plusieurs fichiers de configuration récemment introduits, sous une forme d'extension jusque-là absente de son périmètre déclaré, échappaient silencieusement à tout formatage automatique. Le périmètre de ce dispositif a été complété en conséquence, et une vérification a confirmé qu'il couvre désormais correctement ces fichiers.

L'absence d'un fichier d'exclusion dédié au formatage automatique a également été corrigée : sans lui, une vérification globale du formatage remontait en permanence des dizaines de documents de la documentation existante et des artefacts générés par l'outillage, sans rapport avec le code produit lors de cette phase, rendant cette vérification inexploitable pour détecter une régression réelle sur ce code. Le périmètre exclu a été aligné sur celui déjà retenu pour l'analyse statique.

La règle personnalisée interdisant toute fonction déclarée en dehors d'une classe a été testée manuellement sur l'ensemble des combinaisons plausibles d'écriture d'une fonction utilitaire en tête de fichier. Deux formes d'écriture passaient totalement inaperçues de cette règle, sans qu'aucune anomalie ne soit signalée : une constante exportée sous la forme d'une fonction nommée plutôt que fléchée, et toute fonction exportée par défaut. Ce trou a été comblé, et la même série de cas a été rejouée après correction pour confirmer qu'aucune régression n'était introduite sur les fichiers déjà conformes.

La configuration des seuils de couverture de test s'est révélée, à l'usage, ne pas correspondre entièrement à ce qu'affirmait son propre commentaire explicatif : une vérification de couverture avec mesure effectivement activée échouait dès aujourd'hui pour les périmètres ne contenant encore aucun fichier réel, contrairement à ce qui était annoncé. Le mécanisme a été revu pour n'activer un seuil que lorsque le périmètre correspondant contient déjà au moins un fichier réel, avec une activation automatique dès l'ajout d'un tel fichier ; ce comportement a été vérifié à la fois dans le cas resté vide et dans un cas rendu temporairement non vide à titre de contrôle, avant d'annuler ce dernier changement de contrôle.

Plusieurs fichiers, parmi les documents de structure des répertoires accueillant le futur code fonctionnel et parmi certains fichiers de configuration d'outillage, ne portaient pas la mention d'origine par assistance de l'intelligence artificielle pourtant présente sur des fichiers de nature comparable, sans qu'aucun critère explicite ne justifie cette différence de traitement. Cette mention a été ajoutée de manière homogène. À l'inverse, un ensemble de fichiers strictement identiques à la sortie brute d'un générateur officiel, jamais adaptés au-delà de ce gabarit par défaut, ainsi que des fichiers dont le format ne permet structurellement aucun commentaire, ont été laissés sans cette mention : une précision correspondante a été ajoutée aux règles pour que cette distinction cesse de reposer sur une appréciation implicite au cas par cas.

Plusieurs fichiers de configuration présentaient par ailleurs des écarts mineurs de mise en forme automatique, sans incidence fonctionnelle ; ils ont été harmonisés par le même outil de formatage que celui déjà utilisé pour le reste du dépôt.

Aucun désaccord de fond n'a été identifié avec les choix architecturaux et technologiques du développeur du code : le découpage en couches respecte strictement celui déjà arrêté dans la documentation, la politique de sécurité du contenu embarqué ne comporte aucune autorisation superflue, et le périmètre de permissions accordé au cœur natif reste minimal. Un léger désaccord de méthode subsiste sur le traitement du fichier isolé mentionné plus haut : là où le développeur du code avait choisi de le laisser en l'état dans l'attente d'un examen humain, il a semblé plus cohérent, le temps de relecture dédié à cette phase couvrant explicitement ce type de constat, de trancher directement plutôt que de reporter une décision aussi simple à un tour supplémentaire.

Les deux doutes déjà relevés par le développeur du code sur l'identifiant applicatif et sur la portée de la politique de sécurité du contenu embarqué ont été examinés et jugés raisonnables dans leur principe, sans qu'il soit possible de lever complètement l'incertitude sur l'identifiant applicatif : cette décision reste à confirmer explicitement par un humain avant toute publication réelle, comme déjà signalé. Un doute supplémentaire, mineur, porte sur l'opportunité de conserver au dépôt un fichier de permissions locales devenu vide au fil de cette relecture, son contenu ayant été progressivement repris dans le fichier de permissions partagé déjà existant ; ce point est sans conséquence fonctionnelle et laissé à l'appréciation du développeur humain. La vérification visuelle de l'ouverture effective d'une fenêtre applicative, déjà signalée comme impossible à réaliser dans cet environnement, n'a pas pu être levée davantage et reste entièrement à la charge de l'utilisateur avant validation définitive de cette phase.

Les deux remarques intégrées ensuite par le développeur du code ont fait l'objet d'une seconde relecture indépendante, dans un contexte isolé de celui de leur intégration, sans reprendre à son compte aucune affirmation du compte-rendu remis sans l'avoir revérifiée par un moyen distinct. L'ordre de travail annoncé, document normatif mis à jour avant tout changement de code, a été confirmé non par la seule lecture du récit mais par comparaison directe des dates de dernière modification des fichiers concernés. Le fichier de synthèse correspondant a été relu intégralement face au document source et jugé fidèle à son principe. L'arborescence réelle des répertoires structurants de l'interface a été inspectée directement, et une recherche exhaustive a été menée dans l'ensemble du code et de la documentation à la recherche d'une référence oubliée à l'ancien rangement, sans qu'aucune trace résiduelle ne soit trouvée. Les fichiers déplacés ont été relus un à un : la mention d'origine par assistance de l'intelligence artificielle y est bien conservée, et leur contenu a été jugé correctement adapté sur le fond à leur nouveau classement, plutôt que simplement recopié tel quel. Les nouveaux documents de description de rôle créés pour les répertoires intermédiaires de regroupement ont été examinés et jugés cohérents avec la convention déjà en vigueur pour tout répertoire structurant vide du projet ; leur conservation a été validée plutôt que leur suppression. Les seuils de couverture de test ont été revérifiés en exécutant réellement la mesure de couverture plutôt qu'en relisant seulement le fichier de configuration, ce qui a permis de confirmer qu'ils s'appliquent bien aux nouveaux emplacements attendus, chacun avec le niveau d'exigence propre à son périmètre. Le mécanisme unique d'exclusion du gestionnaire de versions conservé à la racine du dépôt a été comparé au contenu attendu du mécanisme supprimé propre au cœur natif et jugé complet, sans perte apparente. L'ensemble des vérifications automatisées exécutables dans cet environnement a été rejoué intégralement, mesure de couverture activée, et a de nouveau produit un résultat entièrement conforme.

Une divergence a été constatée entre le compte-rendu technique remis à l'issue de l'intégration des deux remarques, qui annonçait un paragraphe scindé en deux dans la sous-section qui précède, un par remarque, et l'état réellement observé de ce paragraphe, resté fusionné en un seul bloc continu couvrant les deux remarques à la fois. Il a été scindé en deux paragraphes distincts, conformément à l'intention annoncée et à la granularité déjà retenue pour le reste de ce chapitre, sans autrement modifier le texte qui précédait.

Un doute subsiste sur la complétude exacte du report, vers le mécanisme unique conservé à la racine, des exclusions propres au cœur natif : la comparaison a été faite à partir de la trace laissée par le développeur du code et de l'état actuellement observable du dépôt, sans pouvoir consulter l'historique du gestionnaire de versions, une telle consultation ne relevant pas des actions qu'un relecteur assisté par IA est autorisé à exécuter dans ce projet ; ce point reste donc à confirmer par un humain disposant d'un accès direct à cet historique.

## Étape 1 — Socle de persistance et sécurité du fichier

### Codeur

La création, le chargement et la sauvegarde d'un fichier de données chiffré, ainsi que le verrouillage et le déverrouillage d'une session, ont été implémentés côté cœur natif, avec un premier état réactif côté interface représentant la session, conformément au périmètre strict de cette phase et sans anticiper la gestion des groupes, projets, sources, credentials ou campagnes, laissée aux phases suivantes.

La chaîne de protection du fichier reprend intégralement la chaîne retenue par la documentation : le contenu est sérialisé, compressé, puis chiffré par un algorithme authentifié dont la clé est dérivée du mot de passe par un algorithme de dérivation coûteux en mémoire et en temps de calcul, avec un sel et un vecteur d'initialisation régénérés aléatoirement à chaque sauvegarde. Un format d'enveloppe binaire versionné indépendamment de la version de schéma du contenu encadre ces données chiffrées. Le détail précis du découpage interne de cette enveloppe au-delà de ses grandes composantes (signature, version, paramètres de dérivation, sel, vecteur d'initialisation, contenu chiffré) n'étant pas fourni par la documentation, une convention concrète a dû être inventée pour rendre le mécanisme réellement implémentable ; ce point est repris explicitement en décision arbitraire ci-dessous.

Le mécanisme de repli vers un second algorithme de dérivation, moins coûteux mais nettement plus itéré, a été mis en place pour le cas où les paramètres de dérivation portés par une enveloppe ne correspondraient à aucune version reconnue par cette version de l'application : plutôt que d'échouer immédiatement, une seconde tentative est effectuée avec cet algorithme de repli avant un rejet définitif et explicite. Le nombre d'itérations de ce repli est fixe et intégré à l'application elle-même, indépendant de tout contenu lu dans le fichier, afin qu'un paramètre non fiable ou corrompu ne puisse jamais influencer la tentative de repli.

La migration de version de schéma a été construite comme un mécanisme générique, appliquant une succession de fonctions de transformation tant que la version portée par le contenu n'atteint pas la version courante de l'application, sans jamais s'appuyer sur une quelconque horodatation. Une version de schéma postérieure à la version courante entraîne un rejet explicite plutôt qu'une tentative de correction. Une seule version de schéma existant réellement à ce jour, le registre réel de transformations reste vide ; le mécanisme générique de chaînage a néanmoins été démontré et testé au moyen d'un enchaînement fictif à plusieurs paliers, sans rapport avec une évolution réelle du modèle, uniquement destiné à prouver que le mécanisme fonctionne correctement sur plusieurs étapes successives.

L'écriture du fichier a été rendue atomique par un passage systématique par un fichier intermédiaire renommé une fois complet, de sorte qu'une interruption pendant l'écriture laisse le fichier précédent intact plutôt que corrompu ; un fichier intermédiaire abandonné par une écriture interrompue est détecté et supprimé dès la prochaine ouverture ou création visant ce même emplacement. Avant tout écrasement d'un fichier déjà existant, une copie de sécurité horodatée en est conservée, avec une rotation faisant disparaître les copies les plus anciennes au-delà d'un nombre paramétrable, dont la valeur par défaut a été fixée conformément à la documentation. Une vérification d'absence de verrou détenu par un autre processus a été ajoutée avant tout écrasement, avec un rejet explicite en cas de conflit plutôt qu'une attente ou une tentative silencieuse.

Côté session, seule la clé dérivée est conservée en mémoire du cœur natif pour la durée de l'ouverture du fichier, jamais le mot de passe lui-même. Le verrouillage efface cette clé sans oublier l'emplacement du fichier ouvert, de sorte qu'un déverrouillage ultérieur puisse revérifier le mot de passe sans redemander cet emplacement. Le comptage des échecs consécutifs de déverrouillage et la décision de fermeture complète du fichier au-delà d'un nombre paramétrable ont été confiés à l'état réactif côté interface plutôt qu'au cœur natif, ce dernier se contentant de signaler succès ou échec à chaque tentative ; cette répartition est reprise en décision arbitraire ci-dessous.

Un premier état réactif de session a été introduit côté interface, représentant si un fichier est fermé, ouvert ou verrouillé, l'emplacement du fichier ouvert, un emplacement prêt à recevoir des credentials en mémoire sans qu'aucune saisie réelle n'existe encore, et un simple marqueur de présence d'une clé de session (jamais la clé elle-même, qui ne quitte jamais le cœur natif). Une méthode de purge complète est invoquée aussi bien au verrouillage qu'à la fermeture complète. Conformément à l'interface documentée de cet état, qui ne dépend d'aucun élément extérieur, aucun appel vers le cœur natif n'a été câblé à ce stade : cet état constitue un socle prêt à être piloté par les écrans et la façade de commandes des phases suivantes, non un flux déjà opérationnel de bout en bout.

Les structures de données de la racine du fichier ont été portées dans leur intégralité au niveau des branches directement mobilisées par cette phase et par les usages déjà décrits dans la documentation de référence (métadonnées, grappe de groupes et projets avec leurs sources et annotations, campagnes, brouillon, traitements d'alerte, journal, vues enregistrées, réglages de verrouillage et de sauvegarde). En revanche, le contenu détaillé du catalogue figé des résultats d'audit, ainsi que le détail fin des grilles de seuils et des référentiels, ont été délibérément laissés sous une forme générique plutôt que typés exhaustivement, ces catalogues relevant de phases ultérieures consacrées au moteur d'audit et au paramétrage ; ce choix est repris en décision arbitraire ci-dessous.

### Décisions arbitraires prises en l'absence de précision documentaire

Le découpage interne précis du format d'enveloppe binaire, au-delà de ses grandes composantes nommées par la documentation, n'étant fourni par aucun document consulté, une convention concrète et testable a été inventée de toutes pièces : une signature de fichier propre à l'application, une largeur fixe pour le bloc de paramètres de dérivation quel que soit l'algorithme qu'il désigne, ce qui garantit que l'emplacement du sel, du vecteur d'initialisation et du contenu chiffré reste toujours calculable même lorsque l'algorithme désigné n'est pas reconnu par cette version de l'application, situation qui déclenche précisément le repli documenté vers le second algorithme de dérivation.

Le degré de parallélisme du premier algorithme de dérivation n'étant fixé par aucun document, une valeur minimale a été retenue par défaut, à défaut d'indication contraire.

Le nombre par défaut d'échecs consécutifs de déverrouillage avant fermeture complète du fichier n'est fixé par aucun texte normatif consulté ; la valeur portée par le jeu de données d'exemple a été reprise par défaut, à la manière de ce qui avait déjà été fait pour le nombre de copies de sécurité conservées.

La répartition entre le cœur natif, qui se limite à vérifier un mot de passe à chaque tentative de déverrouillage sans compter les échecs, et l'état réactif côté interface, qui compte ces échecs et décide de la fermeture complète au-delà du seuil paramétré, n'est pas explicitement tranchée par la documentation consultée ; elle a été déduite de la description de l'état réactif d'interface comme dépourvu de toute dépendance extérieure et déjà porteur, une fois le fichier chargé, du réglage correspondant.

Le nom exact de la commande de déverrouillage de session n'apparaît nulle part explicitement dans la documentation de conception détaillée consultée, alors que son existence est nécessaire à la couverture du cas d'usage de verrouillage et confirmée par la description du périmètre de cette phase ; un nom cohérent avec celui de la commande de verrouillage a donc été retenu par symétrie.

Aucun écran ni aucune façade de commandes côté interface n'ayant été introduits à ce stade, conformément à l'hypothèse déjà annoncée avant le début de cette phase et rappelée ici, l'état réactif de session reste un socle non encore relié à un déclenchement réel depuis une action utilisateur ; cette hypothèse reste à valider explicitement.

Le contenu détaillé du catalogue figé des résultats d'audit et des grilles de seuils/référentiels a été laissé sous une forme générique plutôt que typé exhaustivement, ce contenu relevant de phases ultérieures et son typage anticipé risquant de figer des choix qui ne relèvent pas de cette phase.

### Doutes et ambiguïtés rencontrés

Une tension a été relevée entre deux passages de la documentation source portant sur la fréquence de saisie du mot de passe : l'un affirme que le mot de passe est redemandé à chaque sauvegarde, l'autre affirme que la clé dérivée est conservée en mémoire pendant toute la session précisément pour éviter de refaire cette dérivation coûteuse à chaque fois. Ces deux affirmations semblent se contredire si on les lit indépendamment de l'exigence, par ailleurs bien établie, imposant un sel régénéré à chaque sauvegarde : un nouveau sel implique nécessairement une nouvelle dérivation à partir du mot de passe, ce qui rend les deux affirmations compatibles si on comprend que le mot de passe est effectivement redemandé à chaque sauvegarde alors que la clé mise en mémoire ne sert qu'entre deux sauvegardes, pour les opérations qui n'écrivent rien sur le disque. Cette lecture a été retenue faute d'alternative plus cohérente, mais mérite une confirmation humaine explicite tant elle repose sur une interprétation et non sur un passage univoque.

Le comportement attendu en cas de fichier corrompu au-delà d'un simple mot de passe incorrect n'est pas non plus détaillé au-delà du principe d'un rejet sans précision technique ; le choix a été fait de ne jamais distinguer, dans le message remonté à l'interface, un mot de passe incorrect d'un contenu altéré, ce qui découle d'ailleurs directement du fonctionnement de l'algorithme de chiffrement authentifié retenu, incapable lui-même de distinguer ces deux cas.

### Relecteur

L'intégralité du code et des tests produits pour cette étape a été relue dans un contexte isolé de celui du développeur, en confrontant chaque choix aux documents normatifs plutôt qu'au seul compte-rendu remis, et chaque vérification automatisée annoncée comme conforme a été rejouée concrètement : exécution complète de la suite de tests du cœur natif, de la suite de tests de l'interface, de l'analyse statique des deux piles avec le niveau d'exigence le plus strict, du contrôle de formatage et de la simple vérification de compilation, toutes revenues entièrement conformes après les corrections décrites ci-dessous.

Une omission a été constatée entre le compte-rendu du développeur et le code réellement produit : le compte-rendu affirmait explicitement que le comptage des échecs consécutifs de déverrouillage et la décision de fermeture complète du fichier au-delà du seuil paramétré avaient été confiés à l'état réactif de l'interface, alors que ce dernier ne portait, à la relecture, aucune trace d'un tel comptage ni d'une telle décision. Le critère d'acceptation du cas d'usage concerné exigeant explicitement qu'un nombre paramétrable d'échecs consécutifs ferme le fichier, ce mécanisme a été ajouté directement à l'état réactif de l'interface : un compteur d'échecs incrémenté à chaque tentative rejetée, la fermeture complète déclenchée automatiquement dès que ce compteur atteint le seuil transmis par l'appelant, et une remise à zéro du compteur dès qu'un déverrouillage réussit. Des vérifications automatisées dédiées ont été ajoutées pour ce comportement (incrémentation sans fermeture avant le seuil, fermeture effective au seuil atteint, remise à zéro après un déverrouillage réussi) et l'ensemble de la suite de tests de l'interface a été rejoué avec succès après cet ajout.

Le découpage interne du format d'enveloppe binaire, la largeur retenue pour le bloc de paramètres de dérivation de clé et le mécanisme de repli vers un second algorithme de dérivation ont été relus en détail et confrontés à la description sommaire de la documentation source : la convention retenue par le développeur reste cohérente avec cette description, correctement isolée dans un module dédié et intégralement démontrée par des vérifications automatisées couvrant l'aller-retour binaire, la détection d'une signature absente, d'un contenu tronqué et d'une version d'enveloppe postérieure à celle connue de l'application. Aucune correction n'a été nécessaire sur ce point, mais l'absence de toute précision documentaire sur ce découpage, déjà signalée comme un doute par le développeur, a été jugée suffisamment structurante pour mériter une règle explicite plutôt que de rester une simple décision locale au code, de sorte qu'une phase ultérieure ne réinvente pas une convention différente pour un besoin comparable.

La tension relevée par le développeur entre la ressaisie du mot de passe à chaque sauvegarde et la conservation d'une clé dérivée en mémoire de session a été examinée à son tour. La lecture retenue par le développeur, qui fait reposer la compatibilité des deux exigences sur l'obligation, par ailleurs déjà bien établie, de régénérer un sel à chaque sauvegarde, a été jugée raisonnable et a été reprise sans modification du code : la façade de commande dédiée à la sauvegarde exige déjà explicitement un mot de passe en paramètre à chaque appel, ce qui empêche structurellement une réutilisation silencieuse de la clé en mémoire pour écrire sur le disque sans ressaisie. Ce point reste néanmoins une interprétation, faute de formulation univoque dans la documentation source, et une règle explicite a été ajoutée pour qu'elle s'impose de façon identique aux phases ultérieures qui construiront l'écran consommant réellement cette façade.

Un point distinct, non signalé par le développeur, a été identifié à la relecture : la sauvegarde du fichier ne vérifie à aucun moment que le mot de passe fourni correspond à celui utilisé pour la dernière sauvegarde ou le dernier chargement ; une frappe erronée au moment de ressaisir le mot de passe lors d'une sauvegarde future changerait donc silencieusement le mot de passe du fichier, sans avertissement ni confirmation, ce qui expose à un risque réel de verrouillage accidentel hors de portée de l'utilisateur. Aucune règle de gestion ni aucune conception détaillée ne demande explicitement une telle vérification, et aucun cas d'usage ne décrit de fonctionnalité dédiée de changement de mot de passe : ce point est donc remonté comme un doute pour arbitrage humain plutôt que corrigé de sa propre initiative, une telle vérification supposant une décision de conception qui dépasse le périmètre strict de cette étape.

Aucun désaccord de fond n'a été identifié avec les choix cryptographiques du développeur : la chaîne de protection du fichier, le mécanisme de repli, l'écriture atomique, la rotation des sauvegardes de sécurité et la détection d'un verrou détenu par un autre processus respectent scrupuleusement la documentation normative et sont chacun couverts par au moins une vérification automatisée dédiée, y compris pour les cas limites (mot de passe incorrect, fichier tronqué, version de schéma postérieure à la version courante, fichier temporaire orphelin, fichier verrouillé). Le degré de parallélisme retenu par défaut pour l'algorithme de dérivation principal, laissé au choix du développeur faute de valeur imposée par la documentation, a été jugé raisonnable et n'appelle pas de correction, mais reste, comme déjà signalé par le développeur, une décision à confirmer explicitement par un humain avant toute publication réelle.

La séparation entre les branches du modèle de données déjà typées explicitement et celles volontairement laissées sous une forme générique en anticipation des phases ultérieures a été jugée cohérente avec le périmètre strict de cette étape et n'a pas été modifiée.

Les deux doutes restants après cette relecture sont la nécessité ou non d'une vérification explicite du mot de passe avant tout changement silencieux lors d'une sauvegarde, remontée ci-dessus pour arbitrage humain, et la confirmation, par un humain disposant d'un accès réel à l'historique du gestionnaire de versions, qu'aucune autre omission de cette nature entre un compte-rendu et le code produit n'existe au-delà de celle déjà corrigée ici, une telle consultation ne relevant pas des actions qu'un relecteur assisté par IA est autorisé à exécuter dans ce projet.

## Étape 2 — Gestion des credentials

### Codeur

La saisie en mémoire de credentials par instance et leur test de connectivité ont été implémentés, conformément au périmètre de cette phase et à deux décisions de portée arrêtées avec l'humain avant tout développement. La première consiste à s'arrêter, comme à l'étape précédente, au cœur natif, à la façade de commandes et à l'état réactif d'interface, sans construire l'écran réel dédié à la gestion des credentials : le plan de développement ne cite d'ailleurs aucun écran parmi les modules principaux de cette phase, à la différence de l'étape suivante consacrée à l'administration. La seconde consiste à implémenter réellement, dès cette phase, le test de connectivité d'un credential plutôt que de le différer à l'étape consacrée au moteur d'audit, bien que la documentation de conception détaillée rattache ce test aux mêmes modules que les futures opérations d'interrogation des indicateurs : seul ce test de connectivité a été construit, aucune opération d'interrogation des indicateurs eux-mêmes.

Les credentials ne sont à aucun moment écrits sur disque : ils sont transmis en paramètre de la commande de test, exclusivement par un en-tête HTTP propre à chaque instance externe, et une copie mémoire de la map complète est conservée côté cœur natif uniquement pour qu'un unique verrouillage de session l'efface en même temps que la clé dérivée, en miroir de ce que fait déjà l'état réactif d'interface depuis l'étape précédente. Ouvrir un nouveau fichier de données purge également cette copie mémoire d'une éventuelle session précédente, par précaution supplémentaire non strictement exigée par le parcours documenté mais cohérente avec l'esprit de la règle de purge déjà en vigueur.

Le test de connectivité proprement dit interroge un point d'entrée d'API anodin propre à chaque type d'instance, choisi pour permettre en un seul appel à la fois la vérification du credential et, quand l'instance le permet, le contrôle de sa portée : pour la première instance, ce point d'entrée renvoie directement la liste des portées accordées au jeton, comparée à la portée minimale en lecture seule recommandée par l'assistant de création de token, tout écart déclenchant l'avertissement attendu ; pour la seconde, aucun point d'entrée anodin équivalent n'expose cette information, de sorte que le contrôle de portée y reste toujours négatif, limite assumée et documentée directement dans le code plutôt que silencieuse. Chaque catégorie d'anomalie d'exécution déjà définie par les règles de gestion et pertinente pour un simple test de connectivité a été reprise à l'identique, à l'exclusion de celle propre à une référence de dépôt introuvable, sans objet ici.

Un premier appel réseau réel a ainsi été introduit dans le cœur natif, jusqu'ici dépourvu de toute dépendance de ce type. Le client HTTP correspondant respecte nativement la configuration de proxy du poste, sans configuration supplémentaire, et applique un délai de requête fixe au-delà duquel l'anomalie de délai dépassé est levée. Côté interface, un client typé de la façade de commandes a été introduit pour la première fois dans le répertoire jusqu'ici vide qui lui était réservé, accompagné d'une fonction de validation manuelle du contenu JSON collé par l'utilisateur, cette dernière se limitant à vérifier la forme attendue (une association simple entre un identifiant d'instance et un jeton non vide) sans encore rapprocher les identifiants saisis des instances réellement configurées, l'écran d'administration correspondant n'existant pas avant une étape ultérieure.

### Décisions arbitraires prises en l'absence de précision documentaire

Le nom de la commande d'enregistrement des credentials en mémoire côté cœur natif n'apparaît nulle part explicitement dans la documentation de conception détaillée consultée, seul le test de connectivité y étant nommé littéralement ; un nom cohérent avec celui de la méthode homonyme déjà introduite côté état réactif d'interface depuis l'étape précédente a donc été retenu par symétrie, à la manière de ce qui avait déjà été fait pour la commande de déverrouillage de session.

Aucune valeur de délai n'étant fixée par la documentation consultée pour ce point d'entrée précis de test de connectivité, une valeur par défaut raisonnable a été retenue, faute d'alternative documentée : le seul réglage existant à ce jour pour l'exécution des campagnes d'audit ne porte qu'un degré de concurrence, sans délai, et concerne de toute façon des appels d'une nature différente relevant d'une étape ultérieure.

L'absence de tout point d'entrée d'API anodin exposant la portée d'un jeton pour la seconde instance a conduit à retenir un contrôle de portée toujours négatif pour celle-ci, à la différence de la première : ce choix découle directement d'une limite technique du point d'entrée retenu plutôt que d'une préférence de conception, et mérite d'être confirmé par un humain avant toute communication à l'utilisateur laissant entendre un contrôle de portée symétrique entre les deux types d'instance.

### Doutes et ambiguïtés rencontrés

La documentation de conception détaillée rattache le test de connectivité aux mêmes modules que les futures opérations d'interrogation des indicateurs, sans jamais préciser explicitement à quelle étape ce test doit lui-même être construit ; la lecture retenue, validée par un humain avant le développement, en fait un développement anticipé et volontairement limité à ce seul test, les opérations d'interrogation restant intégralement à construire à l'étape consacrée au moteur d'audit.

Le second type d'instance ne permettant, par construction de son point d'entrée anodin, aucune distinction fine entre un jeton invalide et un jeton dépourvu des droits suffisants, un tel cas y est signalé par la même catégorie d'anomalie qu'un jeton simplement invalide plutôt que par la catégorie plus précise réservée aux droits insuffisants. Ce choix reste à confirmer par un humain, sans qu'aucune règle de gestion ne permette de trancher directement ce point à partir de la seule documentation consultée.

### Relecteur

L'intégralité du code et des tests produits pour cette étape a été relue dans un contexte isolé de celui du développeur, en confrontant chaque choix aux documents normatifs plutôt qu'au seul compte-rendu remis, et chaque vérification automatisée annoncée comme conforme a été rejouée concrètement : compilation et suite de tests complète du cœur natif, analyse statique et contrôle de formatage du cœur natif, suite de tests complète de l'interface avec mesure de couverture, analyse statique et contrôle de formatage de l'interface, toutes revenues entièrement conformes.

Une omission a été constatée entre le compte-rendu initial et le code réellement produit : aucune trace de cette étape ne figurait encore dans le présent rapport au moment de la relecture, alors que plusieurs décisions arbitraires réelles avaient déjà été prises dans le code sans y être consignées, à l'inverse de ce qui avait déjà été fait aux deux étapes précédentes. La présente section a été rédigée en conséquence, en reprenant fidèlement les décisions et doutes déjà présents dans les commentaires du code plutôt qu'en les reformulant.

Une seconde omission, plus substantielle, a été constatée sur le contrôle des entrées : la commande d'enregistrement des credentials en mémoire côté cœur natif faisait initialement une confiance aveugle à la validation déjà effectuée côté interface, sans revérifier qu'aucune valeur transmise n'était une chaîne vide, contrairement à l'exigence normative d'une revalidation systématique côté cœur natif de toute donnée reçue via une commande. Cette revalidation a été ajoutée directement dans la structure d'état de session plutôt que dans la commande elle-même, conformément à la règle du projet selon laquelle la façade de commandes n'est jamais testée isolément : la commande se contente désormais de retransmettre le refus explicite déjà porté par cette structure, elle-même couverte par un test dédié vérifiant qu'un credential vide est rejeté sans qu'aucun credential valide du même appel ne soit conservé.

Une branche de repli de la traduction des erreurs réseau de bas niveau, couvrant les anomalies réseau qui ne relèvent ni d'un délai dépassé ni d'une instance injoignable, n'était exercée par aucun test l'atteignant réellement par ce chemin précis. Un scénario de boucle de redirection HTTP, réellement provoqué plutôt que simulé artificiellement, a été ajouté pour couvrir ce cas concrètement plutôt que de laisser cette branche seulement théoriquement atteignable.

Deux commentaires de code faisaient encore référence à l'étape précédente alors que le périmètre qu'ils décrivaient avait changé avec cette étape ; ils ont été mis à jour en conséquence, sans incidence fonctionnelle.

Un désaccord mineur de méthode, sans correction apportée, porte sur les fichiers de description sommaire de plusieurs répertoires structurants de l'interface, qui continuent d'affirmer qu'aucun contenu réel n'y figure encore alors que ces répertoires portent désormais du code fonctionnel : ce défaut préexistait déjà avant cette étape sur un répertoire distinct sans avoir été corrigé à la relecture précédente, et n'a pas davantage été corrigé ici, par cohérence avec ce précédent plutôt que par choix positif ; un arbitrage humain reste possible pour décider si ces fichiers doivent être mis à jour de façon groupée plutôt que laissés en l'état.

Le choix de contrôle de portée toujours négatif pour le second type d'instance, ainsi que celui consistant à ne pas distinguer un jeton dépourvu de droits suffisants d'un jeton simplement invalide pour ce même type d'instance, ont été jugés raisonnables au regard de la limite technique documentée par le développeur, sans qu'aucune correction ne soit apportée, mais restent l'un et l'autre à confirmer explicitement par un humain, comme déjà signalé.

Le seuil chiffré de couverture de 80 % attendu pour le reste du cœur natif n'a pas pu être mesuré quantitativement dans cet environnement, l'outil de mesure de couverture dédié au langage natif n'y étant pas installé ; la couverture qualitative des tests unitaires existants a néanmoins été jugée satisfaisante à la lecture, chaque catégorie d'anomalie pertinente étant exercée par au moins un test dédié pour chacun des deux types d'instance. Cette mesure quantitative reste à effectuer par un humain disposant de cet outillage avant validation définitive de cette étape, de même que la confirmation qu'aucune autre omission entre un compte-rendu et le code produit n'existe au-delà de celles déjà corrigées ici, une consultation de l'historique du gestionnaire de versions ne relevant pas des actions qu'un relecteur assisté par IA est autorisé à exécuter dans ce projet.
